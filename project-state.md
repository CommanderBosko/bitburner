## Recent Decisions

- **Scoped (via `/interview`) and built purchased-server automation + a dispatch-model rework, chosen via `/research` rather than assumed** (this session): the ask was "automate cloud servers and spread weaken/grow/hack across them for the highest money/sec." Five sources (Bitburner hacking-algorithms docs, a Steam HWGW-manager guide, a Steam batching discussion thread, and two community GitHub script repos) were reviewed in parallel and converged on a three-tier picture: (1) naive single-script-type-per-cycle (what `controller.ts` did before this session) is the weakest, leaving most RAM idle between launches; (2) **proportional/ratio-tuned simultaneous dispatch** (weaken+grow+hack fired together each cycle, thread counts computed off live server state) is the practical sweet spot for early-mid game — RAM-efficient, no precise timing required; (3) **true continuous HWGW batching** (hack→weaken→grow→weaken landing in a precisely 20-200ms-spaced sequence) is the theoretical income ceiling but needs >1TB RAM per target to pay off and was reported fragile in practice by real players — rising hacking level shortens op durations over time and desyncs a precalculated schedule. Decision: built tier 2 for v1 (`controller.ts`'s "Dispatch model" comment block has the same writeup in-code), explicitly deferred tier 3 and multi-target dispatch, and documented both as the next expansion step so a future session doesn't have to redo this research. A second-AI review of the brief (before implementation) caught three real gaps folded into the build: `ns.scp` was needed but unmentioned, weaken sizing needed the *security-side* analysis functions (`growthAnalyzeSecurity`/`hackAnalyzeSecurity`), not just the money-side ones, and the new `server-purchase-manager.ts` needed to launch directly from `controller.ts` (not the `// CHAIN-TAIL` marker) so purchasing starts on `controller.ts`'s first cycle rather than after the whole rest of the boot chain.
- **The actual purchased-server API in this game version is `ns.cloud.*`, not the classic flat `ns.purchaseServer`/`ns.getPurchasedServers`** most community guides (written for older Bitburner versions) reference — confirmed by reading `NetscriptDefinitions.d.ts` directly rather than trusting research-source code samples. `server-purchase-manager.ts` and `controller.ts`'s host-pool logic both use `ns.cloud.getServerNames()`/`getServerLimit()`/`getRamLimit()`/`getServerCost()`/`purchaseServer()`/`getServerUpgradeCost()`/`upgradeServer()`.
- **Reverted the 30-minute Hacknet payback cap** (`2039b97`, prior unclosed session) — it didn't scale with progression: as node costs compounded exponentially, more and more purchases exceeded the flat cutoff and the manager stopped buying anything, stalling income instead of protecting it. Back to unbounded `gain > 0` acceptance; the existing payback-ratio greedy ordering already prioritizes the best available purchase each tick. - **Added self-learning darknet automation** (`65790c7`, prior unclosed session): `darknet-manager.ts` dispatches short-lived `ns.dnet` worker scripts (recon/value/crack) rather than calling `ns.dnet.*` itself, keeping its own resident RAM low; state persists to `/data/darknet-kb.json` with cross-server learned-pattern rankings.
